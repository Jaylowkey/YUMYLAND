# Platform-wide UX, accessibility, and data improvements for YumyLand

This change adds responsive mobile layout to the dashboard, a toast notification system, inline form validation with password strength, real billing API integration, confirmation dialogs for destructive actions, landing page scroll animations, comprehensive ARIA attributes, and SEO metadata. The approach is incremental: each concern is layered onto existing components without restructuring the data layer or auth system.

**Watch for:** Billing API aggregation queries run unfiltered across all companies (confirmed); `window.confirm` for destructive actions bypasses i18n and the project's own Modal component (confirmed); the IntersectionObserver in FeaturesSection never unobserves individual elements after they reveal, meaning the callback fires on every scroll once the section is visible (likely); password validation logic is duplicated between `validateField` and `handleSubmit` in the register page (confirmed).

## High-level view

The dashboard layout is now responsive via a toggle-based sidebar with mobile backdrop overlay and session-driven user info. Both Sidebar and DashboardHeader independently derive the same user state from `useSession()`, duplicating initials logic and logout handlers.

Dashboard CRUD pages (products, categories, reservations) integrate toast feedback but use `window.confirm()` for destructive action confirmation instead of the existing Modal component, producing browser-native dialogs that ignore the user's locale.

The billing page moves from hardcoded mock data to a real Prisma-backed API route. The API runs three separate aggregation queries for summary stats without scoping by company or time range, so the MASTER admin always sees platform-wide totals with no filtering capability.

Landing page animations use CSS `reveal-on-scroll` classes triggered by IntersectionObserver, with staggered entrance delays via inline styles. The accessibility pass adds ARIA labels, roles, and `aria-expanded` attributes throughout, and the Navbar gains a mobile hamburger menu with proper `aria-controls`. Several new pages (customers, billing) contain hardcoded Portuguese strings that bypass the established `next-intl` i18n pattern.

## Billing API without company scoping

The `/api/master/billing` route authenticates via `requireRole("MASTER")` and then runs `prisma.payment.findMany()` and three `prisma.payment.aggregate()` calls without any company filter:

```typescript
prisma.payment.aggregate({
  where: { status: "completed" },
  _sum: { amount: true },
})
```

Every billing page load triggers four queries that scan the entire Payment table. The aggregation isn't scoped to a time range either, so as data grows these queries will degrade. For an MVP admin panel this is functional, but the absence of date filtering or pagination on the aggregation queries is a scaling concern (likely, depends on data volume).

The error handling catches thrown errors by checking `error.message` string contents (`"Unauthorized: No active session"`, `"Requires MASTER role"`). This couples the route to the exact wording of `requireRole`'s error messages. A typed error class or status code would be more robust.

## window.confirm instead of Modal for destructive actions

Products page delete:
```typescript
if (!window.confirm("Tem certeza que deseja excluir este produto?")) return;
```

Reservations status change:
```typescript
if (!window.confirm(`Tem certeza que deseja ${actionLabel} esta reserva?`)) return;
```

The project already has a `Modal` component with Escape key support and animations. Using `window.confirm` bypasses the design system, ignores the user's locale (the strings are hardcoded in Portuguese regardless of the active i18n locale), and produces a browser-native dialog that looks out of place.

## Register page validation duplication

The register page defines validation in two places. `validateField` (a `useCallback`) validates individual fields on blur. `handleSubmit` reimplements the same checks for all fields before submission:

```typescript
// In validateField:
if (value.length > 0 && value.length < 6) { error = "Minimo 6 caracteres"; }

// In handleSubmit:
if (form.password.length < 6) { errors.password = "Minimo 6 caracteres"; }
```

The blur version only flags length if the user has started typing (length > 0), but the submit version flags it unconditionally. This asymmetry means a user who tabs through the password field without typing won't see an error on blur but will see one on submit — a subtle UX inconsistency that will be hard to track down later.

The password strength indicator uses a scoring system (length, uppercase, digits, special chars) that's separate from the validation requirement of "minimum 6 characters." A password of "aaaaaa" passes validation but shows "weak" strength, potentially confusing users who see a successful submit alongside a red strength bar.

## IntersectionObserver lifecycle

```typescript
const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        entry.target.classList.add("revealed");
      }
    });
  },
  { threshold: 0.1, rootMargin: "0px 0px -50px 0px" }
);
```

Once all cards are revealed, the observer keeps watching them and firing callbacks (which are no-ops since the class is already present). Calling `observer.unobserve(entry.target)` inside the `if (entry.isIntersecting)` block would stop unnecessary intersection checks. For 6 feature cards this is negligible, but if the pattern is copied to larger lists it becomes a real concern.

## Hardcoded Portuguese strings outside i18n

The customers page contains untranslated UI text: "Faltam {n} pts", "Nivel maximo", "Pagina X de Y (N clientes)", "Anterior" / "Proximo", "clientes cadastrados". The billing page similarly has "Historico de Cobrancas", "Nenhum pagamento encontrado", "Total Recebido", "Pendente", "Falharam", "Transacoes" hardcoded. The rest of the app uses `next-intl` translations consistently, so these pages diverge from the established pattern and will produce Portuguese-only UI for English-locale users.
